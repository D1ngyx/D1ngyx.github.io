这是一种可行的方法，但是绝对不是一种好的实践方法，因为这不符合设计模式中的 开闭原则。此外，为了方便理解，这个业务逻辑很简单，如果是一个很复杂的业务逻辑，那么将要修改很多地方的代码，这绝对不会是一种好的实践方法。所以这里引出 基于反射的动态代理技术。见 Code-3 。

（四）继续增加业务需求

​ 事情还没完。

​ 在阳光明媚的另一天，你又接到一个业务需求，这个业务的大致内容是，希望你对这个下单业务做一个限流操作，保证用户在一定时间内只能下单一定次数，例如10秒内只能下单3次，防止恶意刷单。

​ 首先分析一下，这个业务逻辑和前面的权限逻辑显然是递进关系的，肯定要先通过权限验证，然后才能下单才会收到限流的限制。实际上的业务逻辑链大概是这样的：业务 -> 权限校验 -> 限流 -> 日志记录 -> 事务管理 -> ...，也就说大多数情况下业务逻辑都具有某种 链式关系 （即便是并列，也能写成链式关系，只是链上的先后顺序无所谓，因为代码执行本质是串行执行，除非涉及到多线程技术，然而这里不讨论多线程技术）。这就引出了 ”代理链“ 模型。

​ 所以在这个业务逻辑中，我们的代理链模型为： 权限校验 -> 限流。 Code-4  直接展示了添加业务逻辑的方法。

（六）动态代理的不同方式

​ 这里提供的方式是基于JDK的内置反射实现的动态代理。实际上还有很多方法，比较知名的是 CGLIB、Javassist，但是这些方法都要用到外部库。

​ Code-3 中，注意到注释中，proxy的参数中，提到了接口一词，proxy的参数之一必须是接口。所以如果业务代码中没有接口的话，编译是会报错的。那么只能使用CGLIB或者其他方式来实现动态代理了。在 Spring 中，就有通过是否存在接口来判断是使用JDK还是CGLIB来实现反射的源码。
